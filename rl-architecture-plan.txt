Thought for 1m 41s

Assuming you mean the classic cart-pole inverted pendulum problem: a cart moves left or right, and the controller must keep the pole upright. In the standard Gymnasium CartPole-v1 version, the observation is a 4D state vector (cart position, cart velocity, pole angle, pole angular velocity), the action space is discrete with two actions {left, right}, the default reward is +1 per step, and an episode terminates if the pole exceeds about ±12°, the cart leaves ±2.4, or the time limit is reached at 500 steps.

How RL controls it

Reinforcement learning treats the pendulum as a sequential decision problem. At each time step, a neural network receives the state, outputs an action distribution, applies an action, and gets a reward. Over many episodes, the policy is updated so actions that keep the pole upright get reinforced and actions that launch the cart into clown-car mode get discouraged. For this kind of low-dimensional control, PPO is a very common choice because it is simple, stable, and uses a clipped objective to prevent overly large policy updates; advantage estimation is typically done with GAE, which reduces gradient variance.

Recommended neural-network architecture for the classic cart-pole

Because the observation is only 4 numbers, you do not need CNNs, transformers, or other ceremonial overkill. A small feed-forward MLP is enough. Stable-Baselines3 documents that for 1D observations, PPO/A2C/DQN commonly use a 2-layer fully connected network with 64 units per layer. Their policy-network guide also shows separate actor and critic heads for cart-pole-style tasks.

A solid architecture is:

Input: 4 floats
s = [x, x_dot, theta, theta_dot]

Shared or separate torso: 2 fully connected layers, 64 units each

Activation: tanh or ReLU

Actor head: 2 outputs → logits for {left, right}

Policy: softmax(logits) gives π(a|s)

Critic head: 1 output → scalar value V(s)

Conceptually:

For cart-pole, I would start with separate actor/critic heads after the shared MLP or with explicitly separate pi=[64,64] and vf=[64,64] branches. That is a clean, boring, reliable choice. Boring is good when the pole is trying to faceplant.

Practical PPO hyperparameter recipe

A very reasonable starting configuration for the classic inverted pendulum is:

Algorithm: PPO

Policy: MlpPolicy

Network: pi=[64,64], vf=[64,64]

Activation: tanh or ReLU

Learning rate: 3e-4

Rollout length (n_steps): 1024 or 2048

Batch size: 64

Epochs per update (n_epochs): 10

Discount factor (gamma): 0.99

GAE lambda: 0.95

Clip range: 0.2

Entropy coefficient (ent_coef): 0.0 to 0.01

Value loss coefficient (vf_coef): 0.5

Gradient clipping (max_grad_norm): 0.5

Parallel envs: 4–8

Training budget: 50k–200k steps is usually plenty for this simple task

Those values line up closely with the documented PPO defaults in Stable-Baselines3: learning_rate=0.0003, n_steps=2048, batch_size=64, n_epochs=10, gamma=0.99, gae_lambda=0.95, clip_range=0.2, ent_coef=0.0, vf_coef=0.5, and max_grad_norm=0.5.

My recommendation is to start here:

policy_kwargs = dict(
    net_arch=dict(pi=[64, 64], vf=[64, 64]),
    activation_fn=torch.nn.Tanh,   # or ReLU
)

Then instantiate PPO with the values above. The 64-64 MLP is right in the sweet spot: small enough to train quickly, large enough to represent the stabilizing policy.

Early in training, actions are nearly random. Over time, PPO pushes the policy toward actions that keep the pole angle near zero while also controlling cart position. The critic learns V(s), an estimate of expected future reward from state s. PPO then updates the actor using the critic’s advantage estimates so the actor increases probability of better-than-expected actions and decreases probability of worse-than-expected ones. That actor-critic structure, combined with clipped updates and GAE, is what makes training much more stable than naive policy gradients.

Training loop in plain English

Reset the environment.

Run the current policy for n_steps, collecting states, actions, rewards, and value estimates.

Compute returns and advantages using GAE.

Update the actor using PPO’s clipped objective.

Update the critic to reduce value prediction error.

Repeat until average episode return approaches the environment ceiling.

Since CartPole-v1 gives +1 per step and caps at 500 steps, “solving” it usually means learning a policy that consistently survives close to the 500-step limit.

Best overall recommendation

If your goal is to teach the classic inverted pendulum with a neural net + RL, I would present it like this:

Use CartPole-v1 as the canonical discrete inverted-pendulum problem.

Use PPO with a 2x64 MLP actor-critic.

Use a categorical policy over left/right actions.

Start with PPO defaults and only tune ent_coef, n_steps, and network size if needed.

Explain that the critic estimates long-term goodness of a state, while the actor learns which corrective push keeps the pole alive.

That gives you something technically sound, easy to explain, and easy to reproduce—unlike half the internet’s RL examples, which are basically “step 1: invoke magic, step 2: publish Medium post.”
